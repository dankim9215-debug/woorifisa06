1.데이터베이스 관리 시스템과 SQL 언어 이해

1-1.데이터베이스 관리 시스템의 이해

- 데이터베이스 관리 시스템(DBMS)은 웹에서의 DB를 다룸
    
- DBMS는 조명 데이터를 관리하기 위해 사용되며, DB를 효과적으로 사용하기 위한 객체들을 관리
    
- DBMS의 키워드들은 데이터베이스 관리 시스템의 구조를 지어내는 작업에 활용
    
- DBMS는 주로 오라클, 마인셀 등의 언어를 사용
    
- 데이터베이스 관리 시스템의 목적은 효과적인 데이터 관리와 사용
    

1-2.SQL 언어의 역할과 DML 이해

- SQL은 데이터베이스 관리 시스템에서 필요로 하는 언어로, 정의를 위한 언어인 DDL과 데이터를 조작하는 DML로 나뉨
    
- DML은 데이터의 조회, 삽입, 수정, 삭제 등에 대해 다룸
    
- DML의 기본적인 활용은 조회, 삽입, 수정, 삭제 등의 명령어로 데이터를 활용하는 것
    
- 데이터 관리에 있어서 DML은 데이터의 본질적인 속성과 특성을 활용
    
- DB를 관리하는 시스템에서 DML은 데이터를 조작하는 기능을 통해 데이터의 변환을 도와줌
    

1-3.DB의 주요 기능과 제어

- DB는 사용자의 아이디를 통해 조회, 삽입, 수정, 삭제 등의 데이터 변경이 가능
    
- DB는 특정 권한을 부여하여 데이터의 보안성을 보장
    
- DB는 특정 시점에 데이터의 상태를 유지하는 기능을 제공
    
- DB 관리 시스템은 전체 DB의 상태를 관리하고, 중요 데이터의 변환을 통제
    
- DB의 주요 기능과 제어는 시스템의 효율적 운영과 데이터의 안정성에 크게 기여함
    

2.데이터베이스 및 관리 시스템 이해

2-1.데이터베이스 관리 시스템의 이해

- 데이터베이스 관리 시스템은 여러 사람이 데이터를 사용할 수 있도록 통합 관리 시스템을 의미함
    
- 시스템은 여러 사용자에게 데이터를 공유하고 관리할 수 있게 돕는 역할을 함
    
- 사용자는 데이터를 생성, 수정, 삭제, 조회 등 다양한 기능을 사용할 수 있음
    
- 관리 시스템은 이 과정들을 효율적으로 처리하고, 데이터의 일관성과 안정성을 유지함
    

2-2.데이터베이스 관리 시스템의 기본적인 원리

- 데이터베이스 관리 시스템은 사용자 데이터의 전송, 저장, 삭제 등을 처리하는 서버 역할을 함
    
- 데이터의 변경 사항이 발생하면 시스템이 이를 자동으로 처리하고, 사용자에게 알림을 보냄
    
- 데이터베이스는 서버에서 운영되며, 이에 필요한 데이터를 서버에서 관리함
    
- 관리 시스템은 다양한 데이터를 관리하며, 사용자에게 필요한 데이터를 제공함
    

2-3.데이터의 성질과 표준화된 데이터 형식

- 데이터의 성질은 자료형에 따라 다르며, 이에 따라 데이터를 저장하고 처리함
    
- 데이터베이스에서는 실수 값 등을 통해 부정 소수점 숫자를 관리함
    
- 데이터의 길이, 수용 가능한 값 등은 표준화된 규칙에 의해 제한됨
    
- 데이터 관리 시스템은 이러한 제한과 규칙을 준수하며 데이터를 저장하고 검증함

3-2.데이터베이스의 세분화된 관리

- 부하가 큰 작업들을 스터디들이에서 먼저 처리하고, 슬레이브 기기를 통해 다른 부하를 조정하는 방식을 제안함
    
- 회사에서 데이터의 생명주기를 마트, 웨어하우스, 레이크 단계로 나눠 관리하는 것을 권장함
    
- 각 단계는 중요한 정보를 다루고, 상황에 따라 유관한 정보를 보호하며 관리하는 데 사용됨
    

3-3.AI 엔지니어링과 데이터베이스 관리

- 사용자로부터 수집된 데이터를 적절한 방법으로 처리하고, 웨어하우스에 저장하는 방식을 권장함
    
- 데이터 웨어하우스에서 데이터를 불러와 관리하고, 이 과정에서 업데이트되는 데이터를 관리하는 방법을 설명함
    
- 이를 통해 효율적인 데이터 관리와 원활한 AI 엔지니어링이 가능함을 강조함
    

4.웨어하우스 데이터 관리와 AI 모델 학습

4-1.웨어하우스와 데이터 관리

- 웨어하우스에서 생성되는 정보는 ETL 과정을 거쳐 빅데이터 시대의 데이터로 전환됨
    
- 데이터는 AR 백업 등의 형태로 실시간으로 수집되며, 필요시 활용됨
    
- 데이터의 생명주기를 거쳐 레이프 단위에서 실제 사용에 이르기까지 관리 필요
    
- RDBMS(연결 데이터베이스 관리 시스템)는 이를 위한 주요한 도구로 활용됨
    

4-2.데이터베이스 매니지먼트 시스템의 변천사

- 초기에는 계층 구조를 기반으로 한 데이터베이스 매니지먼트 시스템이 주로 사용됨
    
- 하지만, 상호관계가 많은 경우 복잡성 증가로 인해 접근 방식의 유연성이 떨어짐
    
- 네트워크 형태의 데이터베이스 매니지먼트 시스템이 등장하며, 유연한 관계 표현이 가능해짐
    
- 하지만, 관계가 복잡해질수록 설계 및 관리가 더욱 복잡해짐
    

4-3.관계형 DB와 AI 모델 학습

- 복잡한 관계를 표현하기 위해 관계형 DB가 등장, 중복을 제거하며 효율성 증가
    
- 관계형 DB는 각 테이블에 고유한 키를 부여하여, AI 모델 학습에 필요한 데이터를 제공함
    
- 복잡한 데이터 관계를 효율적으로 표현하고 추출하는 방식으로 사용됨
    
- 그러나, 빅데이터 시대에 AI 모델 학습은 더욱 복잡한 관계를 효율적으로 관리하기 위해 필요함
    

5.데이터베이스 매니지먼트 시스템과 기술 장점, 단점

5-1.데이터베이스 매니지먼트 시스템의 종류

- C폰씨와 C플 표의 가격을 함께 보여주는 예시로 데이터베이스 매니지먼트 시스템의 중복성 없는 활용 설명
    
- 하나의 테이블로 데이터를 관리하면 중복 데이터를 제거할 수 있음
    
- 테이블마다 먼저 데이터를 변경하면, 해당 테이블의 데이터 변경을 먼저 해야 함
    
- 복잡한 데이터를 하나의 테이블로 관리할 때, 중복된 데이터는 제거되며, 데이터의 무결성은 유지됨
    

5-2.데이터의 관리와 필요성

- 데이터의 관리가 필요한 경우, 데이터의 중복을 제거하는 방법이 필요함
    
- 각각의 관계에 관계를 추가하거나, 데이터의 복잡성을 줄이는 방향으로 나아가야 함
    
- 이에 따라, 정형 데이터, 비관계형 데이터, 다중 데이터 등 다양한 데이터 형태에 맞게 매니지먼트 시스템이 필요함
    
- 예를 들어, 시, 주제, 상품 등의 데이터를 관리하는 방법을 설명함
    

5-3.강의의 장단점과 개선 방향

- 데이터의 복잡성을 줄이기 위해, 데이터를 다루는 방법이 점점 복잡해지고 복잡한 패턴을 이해하도록 도와주는 방향으로 발전함
    
- 하지만, 이러한 방법은 특정 상황에서만 효과적이며, 다른 상황에서는 비효과적일 수 있음
    
- 따라서, 중복성 없는 데이터 관리와 복잡한 데이터의 효율적인 처리 방법에 대한 연구가 필요함
    
- 예를 들어, 고객과 상품 정보를 분리하여 관리하는 방법 등에 대해 설명함

7.데이터베이스 스키마와 샘플

7-1.샘플 DB와 테이블 이용

- ISQL에서 제공한 샘플 데이터베이스와 테이블 이용 방법 설명함
    
- DB 접근 방법 중 하나로 테이블에 최종적으로 접근하거나 기존 명령어 대신 소문자로 접근함
    
- 데이터베이스 스키마를 통해 테이블에 접근하며, 주소공간에 접근하는 방법도 있음
    
- 테이블명으로 접근할 때에는 컬럼을 통해 데이터를 조회함
    
- 국가, 지역, 인구수 등 특정 정보를 찾기 위해 컬럼 사용함
    

7-2.컬럼의 적용과 복잡한 데이터 품질

- 컬럼에 따라 데이터의 프라이머리티를 정하고, 이를 통해 데이터 관리가 가능함
    
- 1부터 2026까지의 작은 규모의 월드 데이터를 복잡한 스키마에 묶어서 관리함
    
- 컬럼명은 특정 키로 관리되며, 이를 통해 특정 정보를 가진 데이터를 정확하게 찾을 수 있음
    
- 컬럼에는 각각 다른 프라이머리티가 부여되며, 이를 통해 데이터의 품질을 확보함
    

7-3.복잡한 테이블의 관리와 특징

- 사케라는 테이블이 있으며, 컬럼은 프라이머리티를 가지지 않으며, 여러 테이블을 합쳐놓은 가상의 테이블임
    
- 액터 인포를 통해 테이블이 없는 경우에 각 테이블을 조회함
    
- 액터 인포를 통해 테이블을 조회하는 방식은 길지만, 필요에 따라 시간을 절약하고 편리하게 데이터를 조회할 수 있음
    
- 사케라는 테이블은 영화, 배우, 장르 등을 묶어놓은 컬럼이 있음
    
- 이 컬럼들은 영화와 배우, 장르 등을 구분하는 가식적인 키로 작동함
    

8.데이터베이스와 테이블 조작에 대한 이해

8-1.데이터베이스와 테이블의 기본 이해

- 데이터베이스에서 테이블은 데이터의 주요 단위임
    
- 미국의 비디오 가게 체인에서 비디오와 가게 정보를 테이블로 정리함
    
- 테이블을 정리하면 필요로 하는 정보들을 가지고 있음
    
- 필름 인스탁은 테이블을 순차적으로 돌려 결과를 받는 프로시저임
    
- 필름 인스탁을 통해 영화의 번호와 해당 가게에 있는 영화의 수를 알아냄
    

8-2.스토어드 프로시저와 펑션의 역할

- 스토어드 프로시저는 테이블의 정보를 순차적으로 돌려 결과를 받음
    
- 펑션은 커리문 안에서 특정 동작을 실행하고 결과를 메모리에 저장함
    
- 펑션은 커리문 안에서 동작을 수행하고 결과를 메모리에 돌려줄 수 있음
    
- 스토어드 프로시저와 펑션은 비슷하지만, 결과의 출력 방식에서 차이가 있음
    
- 펑션은 결과를 직접 메모리에 돌려주며, 스토어드 프로시저는 메모리에 결과를 저장함
    

8-3.테이블의 활용과 테이블에 대한 기본 이해

- 데이터베이스에서 테이블 외에 다른 객체들도 존재함
    
- 데이터의 정렬을 위한 콜레이션에 대해 설명함
    
- 유저 간의 데이터 협업을 위한 콜레이션 이해가 필요함
    
- 데이터 베이스를 만드는 과정에서 발생하는 문제점과 해결 방안을 고려함
    
- 테이블에 대한 기본 이해를 바탕으로 실제 데이터 베이스 활용을 추구함
    

9.데이터베이스 테이블 정렬

9-1.데이터베이스 테이블 정렬 방식 이해

- 데이터베이스 테이블 정렬 시 UTS8, MB4, 제너럴 CI 방식 사용함
    
- 회사에 따라 구 방식 기기들을 유지하는 경우도 있음
    
- 정렬 방식에 따라 데이터를 A, B 등 알파벳 순서로 정렬하거나 대소문자순서로 정렬할 수 있음
    
- 일반적으로는 UTS8 방식으로 정렬하지만, 제너럴 CI 방식도 사용됨
    
- 테이블 생성 시 기본 정렬 방식과 캐릭터셋을 지정함
    

9-2.테이블 정렬과 제약 조건

- 테이블 정렬 시 대소문자, 숫자로 시작하는 컬럼은 제약 조건을 통해 다름을 유지함
    
- 숫자로 시작하는 컬럼명은 헷갈리지 않도록 한글이나 알파벳 소문자로 시작하는 것을 권장함
    
- 제약 조건을 통해 중복되는 값이 없도록 함
    
- 중복되는 값이 있으면 안 되며, 중복을 피하기 위해 절대 중복되는 값이 없도록 함
    
- 컬럼의 자료형에 제약 조건을 넣어 여러 값이 없도록 함
    

9-3.테이블 생성 과정

- 테이블 생성 시 기본 정렬 방식과 캐릭터셋을 지정함
    
- 테이블의 기본 캐릭터셋은 디폴트 캐릭터셋으로 기본값을 만듦
    
- 필요한 경우, 기본키 컬럼을 추가하거나 조합키를 설정할 수 있음
    
- 데이터베이스 단위로 작성 시, 기본 정렬 방식과 캐릭터셋을 지정함
    
- 예외가 있을 경우, 테이블을 생성하여 정렬 방식을 지정할 수 있음
    

10.테이블 정의

10-1.테이블 명세서

- 테이블을 생성, 수정, 삭제하는 명령어에 대한 이해가 필요함
    
- 명령어로 테이블 생성, 수정, 삭제를 할 때 데이터베이스 이름, 테이블 이름, 컬럼명을 정의함
    
- 컬럼명을 정할 때 기본 키를 포함하고 중복되지 않는 값으로 설정하는 것이 중요함
    
- 예를 들어, '예약어' 테이블을 생성할 때 '예약'이라는 단어를 포함시킴
    
- 테이블에 기본 제약 조건을 설정하여 데이터의 무결성을 보장함
    

10-2.제약 조건

- 제약 조건을 설정하여 데이터의 일관성과 무결성을 확보함
    
- '국가명' 테이블에서 '국가 코드' 컬럼에 '국가명' 제약 조건을 설정함
    
- 테이블에 특정 값이 없도록 하고, 해당 값이 없을 때는 '프라이버시'를 보호함
    
- '국가명' 테이블에서 '국가 코드' 컬럼에 '국가명' 제약 조건을 설정함
    
- 컬럼에 '국가명'이 포함된 경우, '국가 코드'는 해당 컬럼에 '시작하는 국가' 값이 있으면 설정하지 않음
    

10-3.테이블 예제

11.데이터베이스 정렬

11-1.데이터 정렬 및 표 생성

- 데이터를 본문에 정렬하고, 여러 열을 추가함
    
- 정렬 시 열 추가가 아니라, 데이터의 순서를 바꿈
    
- 셀렉트, 헤드, 세이브, 요, 순서를 사용하여 데이터 조회함
    
- 셀렉트는 데이터를 선택하는 기능, 헤드는 시작 위치를, 세이브는 저장 위치를, 요는 행을, 순서는 열의 순서를 의미함
    
- 본문에 데이터를 입력할 때, 제약 조건을 넣어 데이터의 순서를 보장함
    

11-2.별 사용의 문제점

- 데이터를 조회할 때, 별을 부르면 많은 양의 데이터가 메모리에 남게 됨
    
- 사용자에게 새로운 정보가 추가되거나 변경될 수 있음
    
- 별은 무책임한 방법이며, 사용하지 않는 것이 좋음
    
- 별을 사용할 경우, 데이터의 모든 열을 불러오는 대신, 화이트 또는 시퀀스 아이디로 불러올 수 있음
    
- 별을 부르면 데이터의 구조를 보여주기 때문에, 별을 사용하지 않는 것이 좋음
    

11-3.데이터 정렬의 기술

- 데이터 정렬을 위해, 셀의 컬럼명을 자세히 명시함
    
- 컬럼을 자세히 명시하면, 불러오는 데이터의 개수가 줄어듦
    
- 컬럼의 순서를 조정하여, 필요한 열만 부르는 것이 좋음
    
- 데이터의 구조를 먼저 확인하고, 필요한 열을 제한하여 조회함
    
- 헤드, 세이브, 요, 순서를 사용하여 데이터를 조회하고, 필요한 열을 부름

15.SQL 검색

15-1.SQL의 기초

- SQL은 정규식과 동일한 기능을 하지만 더 정확한 검색이 가능함
    
- 필요한 값들을 특정한 패턴으로 묶어 검색할 수 있음
    
- 비트윈을 사용하면 특정 연간부터 특정 달까지 찾아주세요고, 우수 조건을 넣어 조건절을 만들 수 있음
    
- 언더바는 한 글자, 퍼센트는 0개 이상 포함된 모든 글자를 의미함
    
- 조건에 맞는 값을 찾을 때는 '마이크' 명령어를 사용함
    

15-2.조건과 패턴

- '언더바'는 언더바의 개수만큼 문자열을 의미함
    
- '언더바'로 시작하거나 끝나는 문자열을 찾을 수 있음
    
- 패턴을 찾을 때는 '마이크' 명령어를 사용해 특정 문자열을 검색할 수 있음
    
- '정규식을 이용한다'는 표현은 다양한 패턴을 원하는 값을 찾을 수 있음을 의미함
    
- '마이크' 명령어로 특정 문자열을 찾을 때, 조건을 포함해 전체 조건을 작성할 수 있음
    

15-3.예제

- 영화 제목에 '천년여우'가 포함된 경우를 찾는 예제임
    
- '언더바'로 시작하거나 '언더바'로 끝나는 문자열을 검색할 수 있음
    
- '여우'나 '공'이 포함된 문자열을 찾을 수 있음
    
- '은/는' 조건을 사용해 특정 문자열이 포함된 행을 선택할 수 있음
    
- 'R like' 명령어로 특정 패턴이 포함된 행을 검색할 수 있음
    

16.AND 조건과 조건의 우선순위

16-1.AND 조건의 사용

- AND라는 조건을 사용해 조건을 작성함
    
- 원하는 조건을 찾기 위해 사용하는 방법을 설명함
    
- 조건이 길어질 경우 사용하는 커리의 길이가 길어지는 문제점이 있음
    
- 따라서 조건을 묶어주면 문제를 해결할 수 있음
    

16-2.조건의 우선순위 설정

- 조건의 우선순위를 설정해 조건의 순서를 명확히 함
    
- 키워드에 원래 동작하는 것으로 설정하고, 키워드의 순서를 비틀어 사용하는 경우가 많음
    
- 위치를 설정하면 조건의 우선순위를 명확히 할 수 있음
    

16-3.조건에 대한 예시 분석

- 조건에 대한 예시를 통해 조건의 중요성과 사용 방법을 설명함
    
- 조건에 따라 다른 결과를 볼 수 있으며, 이를 이해하는 것이 중요함

- 조건에 따라 동일한 결과를 얻을 수 있으므로, 조건의 우선순위를 잘 설정하면 효율적인 결과를 얻을 수 있음

17-2.테이블 명령어의 활용

- 테이블 명령어를 사용해 데이터를 집계 및 분석할 수 있음

- '집계' 명령어의 결과를 특정 기준에 따라 다른 형태로 출력할 수 있음

4.테이블 정렬

4-1.조건에 따른 테이블 정렬

- 1차 테이블에서 2차 테이블을 정렬할 때, 1차 테이블에서 조건을 대입하고, 2차 테이블에서는 정렬하는 방식을 설명함
    
- 정렬된 2차 테이블의 결과를 다시 1차 테이블에서 조회하는 과정을 소개함
    
- 정렬 과정에서 필요한 값이 나올 때까지 반복해서 조건을 확인하는 방식을 설명함
    
- 조건에 따라 테이블을 정렬하는 과정과 결과를 보여주는 "수행 계획"을 설명함
    

4-2.테이블 정렬 비용

- 테이블 정렬 과정에서 쿼리 비용이 어떻게 계산되는지 설명함
    
- 복잡한 조건을 포함한 쿼리를 작성할 때, 원본 테이블에서 데이터를 추려내는 것이 가장 효율적임을 강조함
    
- 테이블 스캔을 통해 1차 테이블에서 조건을 대입한 후, 2차 테이블에서 정렬하는 방식을 소개함
    
- 테이블 정렬 시, 비용이 증가할 수 있음을 주의해야 함을 강조함
    
- 동일한 결과를 얻을 수 있더라도, 원본 테이블에서 데이터를 추려내는 것이 더 효율적일 수 있음을 설명함
    

4-3.테이블 정렬 시 유의점

- 1차 테이블에서 많이 추려서 2차 테이블을 정렬하는 방식을 소개함
    
- 테이블 정렬 시, 조건을 포함한 쿼리를 작성할 때, 필요한 값을 찾아낼 때까지 반복해서 조건을 확인하는 방식을 설명함
    
- 동일한 결과가 나올 때까지, 1차 테이블에서 데이터를 추려내는 것이 효율적임을 강조함
    
- 비용이 증가할 수 있음을 언급하며, 결과가 동일한 경우, 원본 테이블에서 데이터를 추려내는 것이 더 효율적일 수 있음을 설명함
    
- 정렬 시, 조건에 따라 다른 테이블을 생성하는 과정을 설명함

- 패딩에 오더바이는 컬럼 순서를 반드시 포함하며, 불완전한 SQL 함수를 가진 임시 이름을 부여
    

6-2.SQL 함수의 종류와 활용

8.데이터 타임

8-1.데이터 타임 기준 설정

- 데이터 수정 시 날짜와 시간을 추적할 때 표준 단위로는 '시분 주'를 사용함
    
- 데이터 변경 시 '시분 주'를 포함해 더 디테일한 시간 단위까지 기록해야 함
    
- 데이터 수정 시간은 디테일한 숫자까지 기록해 '시분 주'까지' 표시를 원함
    
- 데이터 변경 시 표준 방식인 ISO 8601에 따라 포맷팅해야 함
    
- 데이터 수정 시간을 추적할 때는 '시분 주'를 기준으로 함
    

8-2.데이터 타임의 다양한 방법

- 데이터 수정 시간을 추적할 때 '시분 주'나 '시분'을 기준으로 삼아야 함
    
- '시분 주' 기준 시 '시분'은 '001'부터 '003'까지, '시분'은 '001'부터 '006'까지 가능함
    
- '시분 주' 기준으로 데이터를 수정할 때는 '시분 주' 기준으로 결과를 출력해야 함
    
- '시분'을 기준으로 데이터 수정 시 '시분'은 '001'부터 '006'까지, '분'은 '000'부터 '006'까지 가능함
    
- 데이터 수정 시, '시분'을 기준으로 '시분 주'를 포함해 결과를 출력해야 함
    

8-3.데이터 타임의 활용

- 데이터 수정 시, '시분 주' 기준으로 '시분'과 '분'을 포함해 결과를 출력해야 함
    
- 데이터 수정 시간을 추적할 때, '시분 주'를 포함해 결과를 출력해야 함
    
- 데이터 수정 시, '시분'과 '분'을 기준으로 결과를 출력하면 효율적임
    
- 예를 들어, '데이터 수정 시간을 추적할 때'에서 '시분 주'를 포함해 출력해야 함
    
- 데이터 수정 시간을 추적할 때, '시분'을 기준으로 삼아야 함
    

9.데이터 변환

9-1.데이터형 변환 이해

- 다양한 데이터형(숫자, 문자열, 시간)을 소개함
    
- 각 데이터형의 최댓값과 최솟값을 예시로 설명함
    
- 데이터 베이스 테이블의 데이터 크기에 따라 이상치 판단이 필요함
    
- 데이터 변환 시 오류가 발생할 수 있으니 주의해야 함
    
- 데이터 변환 시 메모리 아끼기 방식이므로 기본 소수점은 부정 소수점으로 봄
    

9-2.조건문과 케이스 분석

- SQL의 조건문(==, >, <)과 한국어의 조건문(==, 대소문자=)을 비교함
    
- 데이터 변환 시 오류를 방지하기 위해 케이스 분석이 중요함
    
- 조건의 디테일한 조건을 한번에 처리하려 하지 말고, 한가지 조건에 대해 여러 개의 경우를 고려해 처리해야 함
    
- 예를 들어, "값이 널이면 널이 출력된다"에서 "값이 널이면"으로 이끌어 나가지 않아야 함
    
- SQL의 불교남용을 막기 위해 조건문을 간결하게 만들 필요가 있음
    

9-3.케이스 분석의 중요성

- 데이터 변환 시, 조건에 따라 다른 값을 출력하므로 케이스 분석이 필요함
    
- 한가지 조건에 대해 여러 경우를 고려해 디테일한 조건을 작성해야 함
    
- 예를 들어, "값이 널이면, 불사슬린 값으로 출력된다"에서 "네"가 출력되면 "네"로만 처리해야 함
    
- 데이터 변환 시, 케이스 분석을 통해 다양한 경우를 고려해야 함
    
- 케이스 분석을 통해 오류를 예방하고, 효율적인 데이터 변환을 가능하게 함
    

10.복잡한 조건문

10-1.케이스문 이해

- 조건문을 단순화할 때는 '==' 사용을 추천함
    
- '==?' 조건에 true 값을 대입한 뒤 '==?'로 바꾸어 쓸 수 있음
    
- 복잡한 조건은 '&&' 사용을 추천하며, '&&?'로 보여줌
    
- 조건문은 '&&?'로 변경하여 작성하면 코드가 간결해짐
    
- '&&?'은 '==?'와는 달리 더 엄밀한 조건 검증이 가능함
    

10-2.케이스 부분 조건문

- 케이스 부분은 값의 비교를 위한 것으로, 조건을 여러 개 작성하여 복잡하게 조합함
    
- 한 개의 조건을 작성하여 원하는 값이 있으면 그 조건에 따라 다른 값을 리턴하도록 함
    
- 복잡한 조건은 '&&' 사용을 추천하며, '&&?'로 보여줌
    
- 한 개 이상의 조건을 작성할 경우, '&&?' 사용을 강조함
    
- 조건을 하나씩 추가할 때, 조건 간 논리적인 흐름을 고려하여 작성함
    

10-3.복잡한 조건문 작성

- 복잡한 조건문은 '&&' 사용을 통해 여러 개의 조건을 구성함
    
- 조건 간 논리적인 흐름을 고려하여 '&&?' 사용을 추천함
    
- 복잡한 조건을 구성할 때는 하나의 조건을 작성한 뒤, 다른 조건을 추가함
    
- 복잡한 조건문은 간결하고 명확하게 작성할 수 있도록 여러 조건을 조합함
    
- 조건문을 통해 필요한 값이 있을 경우, 해당 값을 리턴하도록 명시함
